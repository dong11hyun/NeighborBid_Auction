# AWS 런칭 과정 간소화 설명서 (초보자용)

안녕하세요! 이번에 우리가 AWS 런칭 과정을 훨씬 더 쉽게 만들기 위해 추가한 파일들과, 왜 이렇게 바꿨는지 설명해드릴게요.

## 핵심 변경 사항 요약

**"서버에서 직접 요리하지 말고, 미리 만들어진 밀키트를 가져가자!"**

이전 방식은 "서버(주방)"에 가서 재료를 하나하나 손질하고 요리(파일 생성 및 수정)를 해야 했습니다.
변경된 방식은 "집(로컬)"에서 완벽한 밀키트(설정 파일)를 만들어서 가져가는 방식입니다. 서버에서는 그저 "데우기(실행)"만 하면 됩니다.

---

## 1. 새로 추가된 파일들 (우리의 밀키트)

### 1) [docker-compose.prod.yml](file:///c:/NeighborBid_Auction/docker-compose.prod.yml)
- **역할**: "서버 실행 설계도 (실전용)"
- **설명**: 기존 `docker-compose.yml`은 개발용이라 서버에서 쓰려면 내용을 많이 고쳐야 했습니다. 이제는 고칠 필요 없는 **완성된 설계도**를 따로 만들었습니다.
- **초보자 이점**: 서버에서 `vi`나 `nano` 같은 되게 낯선 에디터로 파일을 수정하다 오타 낼 걱정이 사라집니다.

### 2) [Dockerfile.prod](file:///c:/NeighborBid_Auction/Dockerfile.prod)
- **역할**: "프로그램 설치 설명서 (실전용)"
- **설명**: 개발할 때는 `runserver`를 쓰지만, 실전에서는 `daphne`라는 더 튼튼한 친구가 필요합니다. 이 설정을 미리 담아두었습니다.
- **초보자 이점**: 서버에서 일일이 명령어를 쳐서 프로그램을 띄우는 복잡한 과정을 자동화했습니다.

### 3) [nginx.conf](file:///c:/NeighborBid_Auction/nginx.conf)
- **역할**: "경비실 & 안내 데스크 설정"
- **설명**: 사용자가 사이트에 접속(80번 포트)하면 "보안 접속(443번 포트, HTTPS)으로 가세요"라고 안내하고, 이미지 파일은 어디 있는지 알려주는 역할을 합니다.
- **초보자 이점**: Nginx 설정은 정말 복잡해서 오타 하나만 나도 서버가 안 켜집니다. 이걸 미리 완벽하게 작성해서 넣어두었습니다.

## 2. 이렇게 하면 무엇이 좋아지나요?

| 이전 방식 (Before) | 변경된 방식 (After) |
| :--- | :--- |
| 1. 서버에 접속한다. | 1. 서버에 접속한다. |
| 2. `nano docker-compose.yml` 입력 | 2. (파일 수정 없음) |
| 3. 까만 화면에서 코드 50줄을 직접 수정 | 3. (파일 수정 없음) |

| 6. 실행! | 6. **바로 실행!** |

## 3. 앞으로의 진행 순서

1. **내 컴퓨터에서**: 이 파일들을 깃허브(GitHub)에 올립니다 (`git push`).
2. **AWS 서버에서**: `git pull`로 파일들을 받습니다.
3. **명령어 입력**: `docker compose -f docker-compose.prod.yml up -d` 딱 한 줄만 치면 끝납니다!
